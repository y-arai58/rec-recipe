# Context Notes

## プロダクト概要

「今日何食べる？」という毎日の献立決めの認知コストを解決する料理レコメンドアプリ。
カップル・家族が気分や条件に合った料理を手軽に見つけられるようにする。

---

## 設計判断と背景

### 1. レシピ情報を持たない設計

**判断:** 料理名＋タグのみ保持。レシピ（手順・食材量）は持たない。

**理由:**
- このアプリの目的は「何を食べるか決める」こと。作り方の提供ではない。
- レシピ情報を持つと著作権・更新コストが発生する。
- シンプルなデータ構造にすることで開発・維持コストを下げる。

---

### 2. 認証なし設計

**判断:** ログイン・アカウント機能を持たない。

**理由:**
- 認証は摩擦。「パッと開いてすぐ使える」体験を最優先する。
- 履歴・お気に入りなどパーソナライズ機能はスコープ外のため、認証の必要性がない。
- 将来的に履歴保存が必要になればローカルストレージで代替可能。

---

### 3. タグマッチング（ルールベース）によるレコメンド

**判断:** LLM連携ではなく、シンプルなタグスコアリングでレコメンドする。

**理由:**
- 外部API依存なしで動作するためコスト・レイテンシが低い。
- ロジックが透明で調整しやすい（「この条件ならこの料理が出る」が明確）。
- 数百件規模なら全件スコアリングでも十分高速。
- 将来的なAI強化の余地は残す（ベースが固まってから）。

---

### 4. スクレイピングによる初期データ構築

**判断:** 初期の料理データはスクレイピングで取得し、手動タグ付けで整備する。

**理由:**
- 数百件の料理データを手入力するのは非現実的。
- 外部API（Spoonacular等）は日本の家庭料理カバレッジが低い。
- スクレイピング後にAI（ChatGPT等）でタグを一括付与する運用が現実的。

**リスク:** スクレイピング対象サイトの規約確認が必要。初期は個人利用・非公開として構築を進める。

---

### 5. タグ体系の設計方針

以下のカテゴリ軸でタグを設計する：

| カテゴリ軸 | 例 |
|---|---|
| 料理ジャンル（国） | 日本料理、中華料理、洋食、イタリアン、韓国料理 |
| ボリューム感 | ガッツリ系、サッパリ系、軽め |
| ベース食材 | ご飯もの、麺、パン、汁物 |
| 調理時間（概算） | 短め（〜20分）、普通、時間かかる |
| タンパク源 | 肉、魚、豆腐・卵、野菜メイン |
| 季節感 | 夏向き、冬向き、通年 |

タグ数は 50〜100 程度を目安に設計し、1料理あたり 5〜10 タグを付与する。

---

### 6. 質問フロー設計方針

ユーザーに 3〜5 問の簡単な質問をして条件タグを収集し、料理をスコアリングする。

例:
1. 「今日の気分は？」→ ガッツリ / さっぱり / どちらでも
2. 「何系が食べたい？」→ 和食 / 中華 / 洋食 / なんでも
3. 「主食は？」→ ご飯 / 麺 / こだわらない
4. 「調理時間は？」→ 短め / 普通 / じっくり

回答タグと一致する料理を多い順にソートし、上位 3〜5 件を表示する。

---

## 技術スタック方針

| 項目 | 選定 | 理由 |
|---|---|---|
| Framework | Next.js 16 App Router | SSR/SSGで初期表示速度を確保。スマホでのパフォーマンス重視。 |
| DB | PostgreSQL + Prisma | タグ管理・スコアリングにリレーショナルDBが適切。 |
| UI | shadcn/ui + Tailwind v4 | デザインシステム構築の基盤。アース系カラーをトークンで定義。 |
| スクレイピング | Node.js スクリプト（初回のみ） | DB初期データ構築用。本番アプリには含めない。 |
| ホスティング | Vercel | Next.jsとの親和性・無料枠での個人利用。 |

## Google Stitch 用デザインプロンプト

以下のプロンプトを Google Stitch に送付してデザインを生成する:

```
アプリ名: rec-recipe（料理レコメンドアプリ）

概要:
カップル・家族が「今日何食べる？」という献立決めを解決するモバイルファーストのWebアプリ。
気分や条件を選ぶと、家庭料理を複数レコメンドしてくれる。

デザイン方向性:
- スタイル: おしゃれ・グルメ感のある上質なフードアプリ
- カラーパレット: アース系温かみ（ベージュ、テラコッタ、ブラウン、オフホワイト）
- 角丸・余白: 中間（柔らかすぎず、シャープすぎず）
- フォント: 読みやすい日本語対応フォント（Noto Sans JP + モダンな見出し）
- ダークモード: 不要
- 全体感: 食欲をそそる温かみがありつつ、使いやすくシンプルなUI

必要な画面:
1. レコメンド画面（メイン）
   - 上部: アプリロゴ・タイトル
   - 中央: 質問カード（1問ずつ表示、選択式）
   - 下部: 「レコメンドする」ボタン → 料理カードを複数表示
   
2. 料理詳細・タグ確認画面
   - 料理名（大きく）
   - タグ一覧（ピル形式のバッジで表示）
   - 「別の料理を見る」ボタン

スマートフォン（375px〜）を基準にデザインし、PC（1280px）でも自然に見えること。
```
