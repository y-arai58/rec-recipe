# User Stories

## Epic 1: 料理レコメンド

### US-001: 条件に合う料理をレコメンドしてもらう
- **As a** 献立を決めたい家族/カップル
- **I want** 気分や条件の質問に答えると料理を複数提案してもらう
- **So that** 毎日「何食べる？」を考える認知コストをなくせる
- **Priority:** Must
- **Acceptance Criteria:**
  - [ ] 3〜5問の質問に答えられる
  - [ ] 回答完了後に 3〜5 件の料理が表示される
  - [ ] 表示される料理は回答タグに基づいて絞り込まれている
  - [ ] 「もう一度」「別の候補を見る」で再レコメンドできる

---

### US-002: 提案された料理のタグ情報を確認する
- **As a** レコメンドを受けたユーザー
- **I want** 料理名をタップして詳細とタグを確認する
- **So that** 「なぜこの料理が提案されたか」を理解できる
- **Priority:** Should
- **Acceptance Criteria:**
  - [ ] 料理名をタップすると詳細画面が開く
  - [ ] 詳細画面に料理名・タグ一覧が表示される
  - [ ] タグはカテゴリ別に視認しやすく表示される
  - [ ] 「戻る」でレコメンド結果に戻れる

---

## Epic 2: 料理データベース

### US-003: 管理者が料理データを整備する
- **As a** アプリ管理者（開発者自身）
- **I want** スクレイピングで取得した料理データにタグを付与して管理する
- **So that** レコメンドに使える質の高い料理DBを構築できる
- **Priority:** Must
- **Acceptance Criteria:**
  - [ ] スクレイピングスクリプトで料理名を 200 件以上取得できる
  - [ ] 取得したデータをDBにインポートできる
  - [ ] 各料理に複数のタグを付与できる
  - [ ] タグはカテゴリ別に管理できる（ジャンル・ボリューム・ベース食材など）

---

### US-004: 管理者が料理・タグを追加・編集できる
- **As a** アプリ管理者
- **I want** 管理画面（またはCLI/スクリプト）で料理・タグを追加・編集する
- **So that** データを継続的に改善できる
- **Priority:** Could
- **Acceptance Criteria:**
  - [ ] 料理の追加・編集・削除ができる
  - [ ] タグの追加・編集・削除ができる
  - [ ] 料理とタグの紐付けを変更できる

---

## Epic 3: UI・パフォーマンス

### US-005: スマホで快適に使える
- **As a** スマートフォンユーザー
- **I want** スマホ画面で快適にタップ・スクロールできる
- **So that** 外出先や夕食前のわずかな時間でサッと使える
- **Priority:** Must
- **Acceptance Criteria:**
  - [ ] 375px 幅のスマホ画面でレイアウトが崩れない
  - [ ] タップターゲットが 44px 以上確保されている
  - [ ] 初期表示が 3 秒以内に完了する（3G相当の回線）
  - [ ] PC（1280px）でも自然に表示される

---

### US-006: 質問フローがわかりやすい
- **As a** 初めて使うユーザー
- **I want** 説明なしで直感的に操作できる
- **So that** アプリの使い方を覚えなくても即座に使える
- **Priority:** Must
- **Acceptance Criteria:**
  - [ ] 質問が 1 問ずつ表示される（一度に詰め込まない）
  - [ ] 選択肢がボタン形式で表示され、テキスト入力が不要
  - [ ] 進捗（何問中何問目）が表示される
  - [ ] 選択後に次の質問へ自動遷移する
