# Anti-AI Design Guide

See @references/before-after-examples.md for concrete code examples.

> AI生成UIはたたき台としては優秀だが、重要なのはAIに作らせたものをそのまま採用しないこと。
> 人が意図を持って編集する。このスキルはその「意図」を構造化したもの。

## ❌ AI生成っぽく見えるUIの5つの特徴

### 特徴1: 全部がきれいに整いすぎている
- 破綻しない平均点なデザイン
- 全体的に無難に整っている一方で、優先順位のメリハリが弱い
- 全要素が同じサイズ、同じ余白、同じ色の濃さで並んでいる

### 特徴2: テンプレ感が強い
- カードやセクションなど、よくあるUIパターンの組み合わせ
- 「情報設計から導かれた」のではなく「UIパターンを機械的に当てはめた」印象
- 他のSaaSで見たことがあるような既視感のあるレイアウト

### 特徴3: 色の意味が整理されていない
- 見た目として映える配色はできているが、意味を持った色設計が甘い
- 視覚的には華やかだが、色の役割やルールが伝わりにくい
- 同じ青がCTAにもリンクにもラベルにも使われている

### 特徴4: 本物の運用感が足りない
- 理想的で綺麗なモックデータだけで作られている
- 長文、例外、欠損、異常値、重複などの実運用パターンが想定されていない
- 「名前: 山田太郎」のような完璧なデータでしか見ていない

### 特徴5: 仕上げの削ぎ落としが足りない
- 情報も装飾もやや多い。なくてもよい線や面が残っている
- 「便利そうだから全部載せた」状態
- 足りないより盛りがち

## ✅ 改善アクション（5原則）

### 原則1: 優先順位を明確にする
**Before**: 全要素が同じサイズ・同じ余白で均等に並ぶ
**After**: 主役が大きく、補助情報が小さく、CTAが目立つ

実装:
- 画面の主役を1つに絞る → そこに最大のサイズ・最濃の色・最大の余白を与える
- 「最初に見る → 次に比較する → 最後に行動する」の視線フロー
- text-3xl + font-bold で主役。text-sm + text-muted-foreground で補助
- 意図的な余白の差: 主役の上下に py-8、補助の上下に py-2

```tsx
// ❌ 全部均等
<div className="grid grid-cols-3 gap-4">
  <Card className="p-4"><h3 className="text-lg">A</h3></Card>
  <Card className="p-4"><h3 className="text-lg">B</h3></Card>
  <Card className="p-4"><h3 className="text-lg">C</h3></Card>
</div>

// ✅ 主役を明確に
<div className="space-y-6">
  <Card className="p-8 border-primary">
    <h2 className="text-2xl font-bold">主役の情報</h2>
    <p className="text-lg text-muted-foreground mt-2">補足説明</p>
  </Card>
  <div className="grid grid-cols-2 gap-3">
    <Card className="p-4"><h3 className="text-sm font-medium">補助A</h3></Card>
    <Card className="p-4"><h3 className="text-sm font-medium">補助B</h3></Card>
  </div>
</div>
```

### 原則2: 文脈から設計する
**Before**:「ダッシュボードだからカード3列 + テーブル + グラフ」
**After**: 「この業務で最初に見たいのはXだから、Xを最上部に大きく配置」

実装:
- レイアウトの前に問う:「このページでユーザーは何を達成したいか？」
- そのプロダクトならではの情報単位を先に決める
- 「他SaaSで見た配置だからこれでいい」は禁止

### 原則3: 色の役割を整理する
**Before**: 青をCTAにもリンクにもバッジにも使う
**After**: 色ごとに1つの役割

実装:
```
ブランド色(primary): CTAボタン、選択状態
操作色(secondary): 二次アクション
状態色:
  - success(green): 成功、正常、有効
  - warning(amber): 注意、要確認
  - destructive(red): 危険、エラー、削除
中立色(muted): 非アクティブ、補助テキスト、ボーダー
```

チェック: 全画面を見渡して「この色は何を意味するか」が即座に説明できるか？

### 原則4: 実運用を想定する
**Before**: 「名前: 山田太郎」「金額: ¥1,000」で確認して完了
**After**: 以下の全ケースで破綻しないことを検証

必須テストケース:
```
- テキスト3行以上: 「株式会社〇〇〇〇〇〇〇〇〇〇〇〇〇〇〇〇〇〇」
- テキスト0文字: ""（空文字）
- データ0件: EmptyState が表示されるか
- 数値1桁 vs 6桁: ¥1 vs ¥999,999 でレイアウトが崩れないか
- 名前が極端に長い: 「Superlongfirstnamewithoutspaces Verylonglastname」
- 同一値の重複: 同じ名前のユーザーが3人いた場合
- null/undefined: 欠損データでクラッシュしないか
- 日付のフォーマット差: 2024/01/01 vs 2024-01-01 vs January 1, 2024
```

```tsx
// ❌ 長さを考慮していない
<span>{user.name}</span>

// ✅ truncate + tooltip
<span className="truncate max-w-[200px]" title={user.name}>
  {user.name}
</span>
```

### 原則5: 最後に削る
**Before**: 思いついた要素を全部入れた状態
**After**: 「これがなくても画面の目的は達成できるか？」でYesなら削除

実装:
- 完成後に1要素ずつ「なくても困らないか」を問う
- ボーダーライン → Divider 本当に必要？
- サブテキスト → 本文だけで伝わらないか？
- アイコン → テキストだけで意味が通じないか？
- バッジの色 → 2色で足りないか？
- 装飾的なグラデーション、影 → なくても機能するか？

## セルフレビュー用チェックリスト

UI生成後、コミット前に必ず実行:

```
✅ 主役となる情報が明確か
✅ 情報の優先順位にメリハリがあるか
✅ よくあるUIパターンの寄せ集めになっていないか
✅ そのプロダクトならではの情報設計になっているか
✅ 色に役割とルールがあるか
✅ 同じ色に複数の意味を持たせていないか
✅ 実データを入れても破綻しないか
✅ 長文、欠損、異常値、重複などを想定できているか
✅ 情報や装飾が増えすぎていないか
✅ 最後に削る視点で見直せているか
```

## プロンプト（内部適用）

ページやコンポーネントを生成するとき、内部で以下を常に適用する:

```
以下を考慮してAIっぽさのないUIを作る:
- 情報の優先順位が明確。画面の主役が一目でわかる
- UIパターンの機械的な適用ではなく、業務に合った情報設計
- 色は役割で設計（ブランド色・操作色・状態色・注意色）
- 実運用想定（長文・0件・異常値・重複で破綻しない）
- 最後に削ぎ落としたような完成度
```
